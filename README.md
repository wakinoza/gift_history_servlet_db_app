# 🎁 シンプルな頂き物管理 Webアプリケーション（DB永続化対応版）

## アプリケーション概要

本アプリケーションは、Java Servlet / JSP を用いて開発された
頂き物情報を記録・管理するWebアプリケーションです。

本リポジトリのアプリケーションは、
以前作成した Servlet ベースのアプリケーションを改良し、
データベース（MySQL）による永続化を実現したバージョンです。

画面構成やユーザー操作といった外部的な振る舞いは変更せず、
内部実装を中心に改修を行い、サーバー再起動後もデータが保持される
実用的なWebアプリケーションへと進化させました。

## 主な機能

※ ユーザーから見た機能は、前作と同一です。

1.  ログイン認証 : ユーザー名とパスワードによる簡易認証を実装しています
（学習目的のため、セキュリティは簡易的な構成です）

2. ユーザー情報表示 : ログイン中のユーザー名とログアウトリンクを常時表示します

3. 頂き物の新規記録 : 日付・贈り主・内容など複数項目を入力して記録を作成できます。未入力項目は内部処理で自動補完されます

4. 記録情報の一覧表示（TODO機能）: 記録された頂き物を一覧表示し、「返礼が必要なもの」を明示的に表示することでTODO管理としても機能します

5. 詳細確認・編集・削除 : 記録内容の詳細を表示する画面から、ワンクリックでの返礼済みの更新が可能です

7. 不要データの即時削除　: すべての操作は DBと連動して行われ、登録・更新・削除結果は即座に画面へ反映されます

## 開発背景と位置づけ

本アプリは 個人開発プロジェクトの改良版で、自作アプリケーションへの DB組み込みは2回目の試みです。

DB接続や設定自体は前回の経験を活かし、比較的スムーズに実装できました

一方で、既存ロジックをDB前提に置き換える工程では、
前作以上に多くの修正が必要となりました

## 使用技術

言語：Java SE 21

Web技術：Servlet / JSP / JSTL

DB：MySQL 8.4

開発環境：Eclipse 2025-09

Webコンテナ：Apache Tomcat 10

品質管理：SpotBugs / Checkstyle（Google Checks ベース）

バージョン管理：Git / GitHub

DB接続情報は context.xml に集約し、
アプリケーションコードから分離することで
保守性・再利用性の高い構成を意識しました。

## 採用アーキテクチャ

本プロジェクトは MVC（Model–View–Controller）パターンを採用しています。

1. Controller（制御層）
Servlet がリクエストを受け取り、
Model の処理結果を JSP へフォワードします

2. Model（業務ロジック層）
JavaBeans（Entity）および
ロジッククラス・DAO により、
DBアクセスと業務処理を分離して実装しています

3. View（表示層）
JSP + JSTL を用いて、
Model から受け取ったデータを動的に描画します

この構成により、
画面・ロジック・データ管理の責務を明確に分離しています。


## 開発を通じて得られた学び
1. DB永続化を前提とした設計の重要性

前作ではアプリケーションスコープでデータを管理していたため、
DB導入にあたり 多くのロジック修正が必要となりました。

特に、

- メソッドの責務が曖昧

- 名前から意図が読み取りづらいメソッド

といった点が、修正コストを高める要因となりました。

この経験から、
将来的な拡張や修正を見据えたメソッド設計・命名の重要性を
強く意識するようになりました。

2. ロジックの複雑化と向き合う設計力

前回DB導入したTODOアプリと比較して、
本アプリは画面遷移や業務ルールが複雑であり、
単純な置き換えでは対応できませんでした。

その中で、

- DBアクセス層（DAO）の切り出し

- ロジッククラスへの責務集約

- Controller の肥大化防止

を意識してリファクタリングを行い、
構造を崩さずにDB化を実現できたことは大きな学びでした。

3. 品質ツールを前提とした開発習慣

SpotBugs / Checkstyle を常時有効にし、
警告を放置せず修正することで、

- バグの早期発見

- 可読性の高いコード維持

を得られました。
